### 问题/bug简述

本机或虚拟机联调时缺少多实例隔离入口，导致无法可靠验证真实双端文件传输。

### 问题详细原因

GUI 全量测试需要至少两个运行端：发送端和接收端。当前本机可见 Hyper-V 组件，但 `Get-VM`、`Get-VMSwitch` 和 `Get-VMNetworkAdapter` 被系统授权策略拒绝，不能直接启动或管理 VM；同时早期项目代码把本地数据目录固定为 `user.home/.lantransfer/<repo>`，把发现端口固定为 `45331`，把接收传输端口固定为 `45332`。如果直接在宿主机多开 JavaFX 实例，多个实例会共享账号、设置、近期对象和接收目录，发现响应线程和接收线程也会互相竞争固定端口；进一步排查时还发现当前 Windows 环境对 `45331-45336` 和临时测试的 `45431-45432` 端口均返回 `Address already in use`，即使 `netstat` 看不到占用进程，导致 `LanPeer` 和 `UdpRx` 后台监听线程启动后立即退出，用户列表自然找不到其他运行端。旧实现里监听线程异常又不够明显，容易出现只测到 UI 流程、没有测到真实双端传输的假通过结果。

### 详细的解决方法

已增加轻量级运行参数作为多端联调入口：`lantransfer.dataDir` / `LANTRANSFER_DATA_DIR` 用于隔离每个实例的账号、设置、近期对象和接收目录；`lantransfer.discoveryPort` / `LANTRANSFER_DISCOVERY_PORT` 用于为每个实例指定独立发现响应端口；`lantransfer.discoveryPorts` / `LANTRANSFER_DISCOVERY_PORTS` 用于告诉扫描端要探测哪些发现端口；`lantransfer.transferPort` / `LANTRANSFER_TRANSFER_PORT` 用于为不同实例指定独立接收端口。默认发现端口已改为 `49131`，默认传输端口已改为 `49132`；`run8.ps1` 的三实例端口也改为 `49131-49136`，这些端口已用系统 UDP 绑定检查验证可用。扫描地址中已加入 `127.0.0.1`，没有 VM 管理权限时也可以在宿主机上启动两个或三个真实 JavaFX 进程做端到端传输测试；有 VM 时仍可使用默认发现端口，或按 VM/宿主隔离需求显式指定端口。`LanPeer` 和 `UdpRx` 后台监听线程退出时已改为打印异常到错误日志，后续端口占用、权限或协议异常能从启动日志直接定位。

### 问题/bug简述

局域网扫描遇到单个广播地址发送失败时会提前结束，导致已可达设备没有进入用户列表。

### 问题详细原因

双实例测试中，接收端 UDP 探针已经能从 `127.0.0.1:20137` 收到 `receiver1` 的发现响应，说明底层发现响应线程正常；但从 GUI 点击“扫描用户”后，发送端用户列表仍回退到 18 个演示用户。原因是 `LanPeer.scan()` 把所有地址/端口的 `socket.send(...)` 和后续接收回复逻辑放在同一个 `try` 中，只要某个网卡广播地址、全局广播地址或端口发送失败，就会进入 catch 并直接返回旧的 `sorted()`，不会继续等待前面已成功发出的发现包回复。

### 详细的解决方法

已把单个发现包发送提取为 `sendDiscover(...)`，每个地址端口发送失败只跳过当前目标，不影响其他地址，也不会阻断后续接收阶段。这样即使某个广播地址被系统网络策略、防火墙、虚拟网卡或 VPN 拒绝，扫描仍能收集其他地址，尤其是本机双实例测试用的 `127.0.0.1` 回复。

### 问题/bug简述

GUI 点击“扫描用户”后，扫描页一直停留在“正在扫描中”，即使设备已经被发现也不会进入完成态。

### 问题详细原因

`Scan.showScanPage()` 原实现先调用 `scanLanDevices()`，等后端扫描完成并返回设备列表之后才渲染页面；但渲染出来的页面标题、状态文案、按钮仍然全部是“正在扫描”和“取消扫描”。也就是说，页面显示出来时扫描其实已经结束，却没有任何“扫描完成”状态、没有“查看用户列表”入口，也没有自动回到用户列表。GUI 双实例测试中雷达上已经出现 `receiver1` 和 `sender1`，但页面仍长时间显示“正在扫描中，请稍候...”，用户只能手动取消，属于前端状态机收束错误。

### 详细的解决方法

已把扫描页拆成两个明确阶段：进入页面时立即渲染扫描中状态并启动后台扫描；扫描完成后渲染“扫描完成”结果页，显示发现用户数量、雷达结果、“查看用户列表”和“重新扫描”按钮。取消扫描时会让当前扫描轮次失效，即使后台扫描稍后返回，也不会把用户已经离开的页面重新拉回扫描结果页。这样扫描功能既能展示等待过程，也能在完成后给用户明确出口。

### 问题/bug简述

接收端切到“忙碌”后，发送端可能在用户确认前提前失败，日志显示“发送请求未确认”。

### 问题详细原因

GUI 双实例测试中，接收端 `receiver1` 已在“我的”页面切换为“忙碌，接收前请确认”，发送端向近期对象 `receiver1` 发送 `gui-transfer.txt` 时，接收端确实弹出了“接收确认”对话框；但发送端约 3 秒内就进入失败结果，统计为成功 0、失败 1、重试 3，日志显示“发送请求未确认”。代码层面原因是 `UdpTx.beginTimeout(...)` 只在发送端缓存的目标对象 `userStatus == BUSY` 时才把 BEGIN 开始包 ACK 等待时间放宽到 15 秒；而近期对象或旧用户列表里的目标状态可能仍是 DEFAULT/ONLINE，接收端实际状态已经变成 BUSY，导致发送端仍按普通 500ms ACK 超时和最大重试次数执行，确认弹窗还没来得及操作就失败。

### 详细的解决方法

已把 `UdpTx` 的文件开始包确认等待统一改为 15 秒，分片 DATA ACK 仍保持 500ms 快速确认和重试。BEGIN 包是唯一可能触发接收端用户交互的协议阶段，因此即使发送端缓存状态不是 BUSY，也要给接收端确认框足够时间；真正的数据分片阶段不涉及弹窗，继续使用短超时保证失败重试效率。`UdpWireCheck` 已新增“发送端目标状态仍为默认、接收端实际忙碌且 2.2 秒后才同意”的自检，旧逻辑会提前失败，新逻辑会等待确认后成功落盘。
