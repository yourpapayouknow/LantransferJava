## 已发现问题

### 问题/bug简述

本机或虚拟机联调时缺少多实例隔离入口，导致无法可靠验证真实双端文件传输。

### 问题详细原因

GUI 全量测试需要至少两个运行端：发送端和接收端。当前本机可见 Hyper-V 组件，但 `Get-VM`、`Get-VMSwitch` 和 `Get-VMNetworkAdapter` 被系统授权策略拒绝，不能直接启动或管理 VM；同时项目代码把本地数据目录固定为 `user.home/.lantransfer/<repo>`，把发现端口固定为 `45331`，把接收传输端口固定为 `45332`。如果直接在宿主机多开 JavaFX 实例，多个实例会共享账号、设置、近期对象和接收目录，发现响应线程和接收线程也会互相竞争固定端口；实际测试中即使传输端口已隔离，发送端用户列表仍只能看到演示用户，看不到接收端 `receiver1`，说明固定发现端口会让多实例扫描结果不可靠，容易出现只测到 UI 流程、没有测到真实双端传输的假通过结果。

### 详细的解决方法

已增加轻量级运行参数作为多端联调入口：`lantransfer.dataDir` / `LANTRANSFER_DATA_DIR` 用于隔离每个实例的账号、设置、近期对象和接收目录；`lantransfer.discoveryPort` / `LANTRANSFER_DISCOVERY_PORT` 用于为每个实例指定独立发现响应端口；`lantransfer.discoveryPorts` / `LANTRANSFER_DISCOVERY_PORTS` 用于告诉扫描端要探测哪些发现端口；`lantransfer.transferPort` / `LANTRANSFER_TRANSFER_PORT` 用于为不同实例指定独立接收端口。扫描地址中已加入 `127.0.0.1`，没有 VM 管理权限时也可以在宿主机上启动两个真实 JavaFX 进程做端到端传输测试；有 VM 时仍可使用默认发现端口，或按 VM/宿主隔离需求显式指定端口。
