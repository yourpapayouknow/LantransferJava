## 已发现问题

### 问题/bug简述

本机或虚拟机联调时缺少多实例隔离入口，导致无法可靠验证真实双端文件传输。

### 问题详细原因

GUI 全量测试需要至少两个运行端：发送端和接收端。当前本机可见 Hyper-V 组件，但 `Get-VM`、`Get-VMSwitch` 和 `Get-VMNetworkAdapter` 被系统授权策略拒绝，不能直接启动或管理 VM；同时项目代码把本地数据目录固定为 `user.home/.lantransfer/<repo>`，把接收传输端口固定为 `45332`。如果直接在宿主机多开 JavaFX 实例，多个实例会共享账号、设置、近期对象和接收目录，并且接收端口互相竞争，容易出现只测到 UI 流程、没有测到真实双端传输的假通过结果。

### 详细的解决方法

已增加轻量级运行参数作为多端联调入口：`lantransfer.dataDir` / `LANTRANSFER_DATA_DIR` 用于隔离每个实例的账号、设置、近期对象和接收目录；`lantransfer.transferPort` / `LANTRANSFER_TRANSFER_PORT` 用于为不同实例指定独立接收端口。后续 GUI 传输测试可以在宿主机和 VM 中分别用独立数据目录与端口启动，也可以在没有 VM 管理权限时在宿主机上启动两个真实 JavaFX 进程，避免端口和数据文件互相污染。
